# エージェントチーム構成（実装記録）

> [[PM・開発者向けAIエージェントチームプラン]] を元に作成した Claude Code サブエージェント定義の記録。
> 定義ファイルの実体はリポジトリの `.claude/agents/`（全17体）。作成日: 2026-07-12

## 使い方

- Claude Code のセッション内で「〜を tech-lead に依頼して」「code-reviewer にレビューさせて」のように役割名で指名すると、対応するサブエージェントに委任される
- すべてのエージェントはタスクボード `AI-Operations/purpose-board/` と共有ドキュメント（この vault）を読み書きの起点にする
- ハンドオフはチャット内で完結させず、ボードのカード更新（From / To / 期待アウトプット / 期限 / リンク）で行う

## チーム一覧

### PMチーム

| エージェント | モデル | 主なツール | 役割の要点 |
|---|---|---|---|
| pm-director | Opus | 読み取りのみ | 価値検証・Go/No-Go観点整理・振り分け案。最終決定はしない |
| requirements-analyst | Sonnet | 読み取りのみ | 曖昧さ検出と質問リスト。仕様は書き直さない |
| risk-reviewer | Opus | 読み取りのみ | リスク一覧（確率×影響度）と軽減策の選択肢。計画は修正しない |
| stakeholder-comms | Sonnet | 読み書き | 週次報告・説明資料のドラフト。送信はしない |
| slide-builder | Sonnet | 読み書き+Bash | テンプレート準拠の顧客向けスライドドラフト（python-pptx） |
| slide-reviewer | Sonnet | 読み取り+Bash+レポート保存 | **PMチーム全成果物のレビュー**（スライド4観点＋その他成果物5観点）。書き直しはしない |

### エンジニアリングチーム

| エージェント | モデル | 主なツール | 役割の要点 |
|---|---|---|---|
| tech-lead | Opus | 読み書き（コードは書かない） | 受付窓口。明確化→**ワークフロー選択（§6.2）**→分解→委任ブリーフ→QA差し戻し→納品サマリー |
| architect | Opus | 読み取りのみ | 設計諮問・トレードオフ評価・設計承認/差し戻し。実装しない |
| backend-dev | Sonnet | フル | 委任ブリーフの指定ワークフロー（OpenSpec / Superpowers）で実装＋テスト（§6.4 準拠） |
| frontend-dev | Sonnet | フル | 同上（UI）。API契約は変更しない |
| qa-engineer | Sonnet | 読み取り+Bash | テスト実行して品質評価。テスト実効性（数字合わせ検出）も見る。修正はしない |
| code-reviewer | Sonnet | 読み取り+Bash | diff レビュー（重要度つき指摘のみ） |
| researcher | Sonnet | 読み取り+Web | ソース付き調査サマリー |
| nightly-qa | Haiku | 読み取り+Bash+レポート保存 | テストスイート実行→`AI-Operations/review-results/` にレポート、失敗時は backlog に起票 |

### インフラ運用チーム

| エージェント | モデル | 主なツール | 役割の要点 |
|---|---|---|---|
| board-keeper | Haiku | 読み書き | ボードのフォーマット統一・放置検出・ステータス矛盾/ワークフロー未記載のフラグ。削除・優先度変更はしない |
| daily-dispatch | Sonnet | フル | ボードから最優先1件のみ処理して書き戻し。人間承認が必要なタスクは着手しない |
| docs-librarian | Haiku | 読み書き | vault への整理・リンク付け・タグ付け。内容は改変しない |

## 段階的導入（プラン §3 準拠）

- **Phase 0**: board-keeper から運用開始（ボードは `purpose-board/` として整備済み）
- **Phase 1**: tech-lead / code-reviewer / requirements-analyst
- **Phase 2**: architect / backend-dev / frontend-dev / qa-engineer ＋ nightly-qa / daily-dispatch の定期実行化
- **Phase 3**: risk-reviewer / stakeholder-comms / slide-builder / slide-reviewer（導入前に顧客説明用テンプレート .pptx の配置とパス固定が必要）

## 運用セットアップの状況（2026-07-12 更新）

- **定期実行（devcontainer 内で完結・推奨）**：コンテナ起動時の `postStartCommand`（`.devcontainer/start-services.sh`）が vault の `AI-Operations/scripts/setup-crontab.sh` を呼び、cron に **nightly-qa（毎晩 23:30）/ daily-dispatch（毎朝 7:00）** を自動登録する。ランナーは `scripts/run-nightly-qa.sh` / `scripts/run-daily-dispatch.sh`（`/src` 起点・ヘッドレス Claude Code を `--permission-mode acceptEdits` で無人実行）、ログは `/tmp/ai-ops-cron/`。前提と制約:
	- Claude Code CLI は `.devcontainer/setup.sh` でインストールされる（`npm install -g @anthropic-ai/claude-code`。既存コンテナではリビルドか手動インストールが必要）
	- **コンテナ内で一度 `claude` を起動してログインしておく**（未ログインならランナーはエラー終了するだけで何も実行されない）。定期実行は Claude 利用量を毎日消費する点に注意
	- cron はコンテナが起動している時間帯しか動かない（停止中に過ぎたジョブの追い掛け実行はしない）
	- Superpowers ワークフローをコンテナ内で使う場合は、コンテナ内の Claude Code で別途 `/plugin install superpowers@superpowers-marketplace` が必要（プラグインはユーザーホーム単位のため）
- **定期実行（Windows ホスト・代替）**：ランナー `scripts/run-nightly-qa.ps1` / `scripts/run-daily-dispatch.ps1` と登録スクリプト `scripts/register-scheduled-tasks.ps1` を準備済み（ログは `logs/` に保存）。登録タスクはヘッドレスの Claude Code を無人実行するため、**登録スクリプトはユーザー自身が内容を確認して実行する**（未登録）。devcontainer 側と二重登録しないこと
- **スライドテンプレート**：`AI-Operations/templates/customer-slide-template.pptx` に配置済み。青×白基調（濃紺 #1F3864 / ブルー #2E75B6 / 淡青 #DEEBF7）・游ゴシック・5レイアウト（タイトル / 中扉 / 1カラム本文 / 2カラム比較 / まとめ、各スライドのノート欄に使い方あり）。再生成は `AI-Operations/scripts/generate-slide-template.py`。`.claude/agents/slide-builder.md` にパス固定済み
- **既存の運用ロール文書**（`AI-Operations/agents/` の dispatcher / engineering-orchestrator / reviewer 等）とサブエージェント定義の統合・整理は別途検討
