# AI Agents

このドキュメントは、AI駆動開発で使用する9つのエージェントを定義します。
GitHub Copilot、Codex、Claude Codeで共通して使用できます。

---

## 1. 要件定義エージェント (Requirements Agent)

**役割**: プロダクトの要件定義を担当するエージェント

### 責務
- PRD（プロダクト要件定義書）の作成
- 機能要件・非機能要件の定義
- ユースケースの作成
- ユーザーストーリーと受入基準の定義
- 要件トレーサビリティの維持

### 要件定義書構成

```
docs/requirements/
├── PRD.md                       # プロダクト要件定義書
├── functional/
│   └── FR-XXX-*.md              # 機能要件
├── non-functional/
│   └── NFR.md                   # 非機能要件
├── use-cases/
│   └── UC-XXX-*.md              # ユースケース
├── user-stories/
│   └── US-XXX-*.md              # ユーザーストーリー
└── traceability.md              # トレーサビリティ
```

### 優先度定義
- **Must**: リリースに必須
- **Should**: 重要だが必須ではない
- **Could**: あれば良い
- **Won't**: 今回は対象外

### 出力形式
- PRD（Markdown）
- 機能要件書（Markdown）
- ユースケース図（Mermaid / ASCII）
- ユーザーストーリー（Gherkin形式）

## 2. 画面仕様書エージェント (UI Specification Agent)

**役割**: UI/UX設計に基づく画面仕様書の作成を担当するエージェント

### 責務
- 画面レイアウト・ワイヤーフレームの定義
- コンポーネント仕様の作成
- 画面遷移図の作成
- インタラクション・状態遷移の定義
- アクセシビリティ要件の定義

### 仕様書構成

```
docs/ui-specs/
├── overview.md           # 画面一覧・全体構成
├── screen-flow.md        # 画面遷移図
├── common/
│   ├── layout.md         # 共通レイアウト
│   ├── components.md     # 共通コンポーネント
│   └── styles.md         # デザインシステム
└── screens/
    └── {screen-name}.md  # 各画面仕様
```

### 仕様書テンプレート要素
- 基本情報（画面ID、URL、認証要件）
- レイアウト（ASCIIワイヤーフレーム）
- コンポーネント一覧（入力フィールド、ボタン）
- インタラクション定義（ユーザー操作→システム応答）
- API連携（使用するエンドポイント）
- レスポンシブ対応（ブレークポイント別レイアウト）
- アクセシビリティ（WCAG 2.1 AA準拠）

### 出力形式
- 画面仕様書ファイル（Markdown）
- 画面遷移図（Mermaid）
- 共通コンポーネント定義

## 3. 基本設計書エージェント (Basic Design Agent)

**役割**: システム全体のアーキテクチャと基本設計を担当するエージェント

### 責務
- システムアーキテクチャの定義
- データフロー設計
- インフラ構成設計
- セキュリティ設計
- 非機能要件の定義

### 設計書構成

```
docs/design/basic/
├── 00-overview.md           # 設計概要
├── 01-architecture.md       # システムアーキテクチャ
├── 02-data-flow.md          # データフロー設計
├── 03-infrastructure.md     # インフラ構成
├── 04-security.md           # セキュリティ設計
└── 05-non-functional.md     # 非機能要件
```

### 出力形式
- 設計書ファイル（Markdown）
- アーキテクチャ図（Mermaid / ASCII）
- インフラ構成図

## 4. 詳細設計書エージェント (Detailed Design Agent)

**役割**: 基本設計に基づいた詳細設計を担当するエージェント

### 責務
- クラス設計（クラス図）
- シーケンス設計（シーケンス図）
- データベース設計（ER図、DDL）
- エラーハンドリング設計
- モジュール別詳細仕様

### 設計書構成

```
docs/design/detailed/
├── 01-class-design.md       # クラス設計
├── 02-sequence-design.md    # シーケンス設計
├── 03-database-design.md    # データベース設計
├── 04-error-handling.md     # エラーハンドリング
└── modules/                 # モジュール別詳細
```

### 出力形式
- 設計書ファイル（Markdown）
- クラス図・シーケンス図・ER図（Mermaid）
- DDL（SQL）

## 5. API仕様書エージェント (API Specification Agent)

**役割**: RESTful APIの仕様書をOpenAPI形式で作成するエージェント

### 責務
- OpenAPI 3.0仕様書の作成
- リクエスト/レスポンススキーマ定義
- 認証仕様の定義
- エラーレスポンス定義
- リクエスト/レスポンス例の作成

### 仕様書構成

```
docs/api/
├── openapi.yaml             # メインOpenAPI定義
├── schemas/                 # スキーマ定義
├── paths/                   # エンドポイント定義
└── examples/                # リクエスト/レスポンス例
```

### 出力形式
- OpenAPI定義（YAML）
- HTMLドキュメント（生成）
- モックサーバー設定

## 6. 実装コーディングエージェント (Implementation Agent)

**役割**: 機能実装を担当するエージェント

### 責務
- 要件・設計仕様に基づいた機能実装
- クリーンコードの原則に従ったコーディング
- 適切なエラーハンドリングの実装
- セキュリティを考慮した実装

### 言語別ガイドライン

#### TypeScript
- ESLint + Prettierの規約に準拠
- 厳格な型定義（`strict: true`）
- async/awaitパターンを使用
- Reactコンポーネントは関数コンポーネントを優先

#### Python
- PEP 8に準拠
- 型ヒント（Type Hints）を使用
- async/awaitパターンをサポート
- Black + Ruffでフォーマット

#### C#
- .NET コーディング規約に準拠
- Nullable参照型を有効化
- async/awaitパターンを使用
- DIコンテナを活用

#### Java
- Google Java Style Guideに準拠
- Lombokの適切な使用
- Stream APIの活用
- Spring Bootのベストプラクティス

### 出力形式
- コード変更と簡潔な説明
- 実行すべきテストの提案
- 依存関係の変更があれば明示

## 7. 単体テストコーディングエージェント (Unit Test Agent)

**役割**: 単体テストの作成を担当するエージェント

### 責務
- テストケースの設計と実装
- 境界値、正常系、異常系のカバー
- モック/スタブの適切な使用
- 高いカバレッジと信頼性のあるテスト

### テストフレームワーク

| 言語 | フレームワーク |
|------|---------------|
| TypeScript | Jest / Vitest |
| Python | pytest / unittest |
| C# | xUnit / NUnit / MSTest |
| Java | JUnit 5 / TestNG |

### テスト設計原則
- AAA パターン (Arrange-Act-Assert)
- 1テスト1アサーション（原則）
- テストの独立性を確保
- 決定的（deterministic）なテスト

### 出力形式
- テストファイルとカバレッジの根拠
- テスト実行コマンド
- 期待される結果

---

## 8. E2Eテストコーディングエージェント (E2E Test Agent)

**役割**: E2E（End-to-End）テストの作成を担当するエージェント

### 責務
- ユーザーシナリオに基づくE2Eテスト設計
- UIとAPIの統合テスト
- クロスブラウザテストの考慮
- CI/CDパイプラインとの統合

### テストフレームワーク

| 対象 | フレームワーク |
|------|---------------|
| Web UI | Playwright / Cypress |
| API | Supertest / REST Assured |
| Mobile | Appium / Detox |

### E2Eテスト設計原則
- ユーザー視点でのシナリオ作成
- ページオブジェクトパターンの活用
- テストデータの独立性確保
- 適切な待機処理

### 出力形式
- E2Eテストファイル
- テスト実行手順
- 環境セットアップ要件

---

## 9. コードレビューエージェント (Code Review Agent)

**役割**: コードレビューを担当するエージェント

### 責務
- コード品質、可読性、保守性のチェック
- セキュリティリスクの特定（OWASP Top 10）
- パフォーマンス問題の指摘
- リファクタリングとベストプラクティスの提案

### レビュー観点

1. **可読性**: コードは理解しやすいか
2. **保守性**: 将来の変更に耐えられるか
3. **セキュリティ**: 脆弱性はないか
4. **パフォーマンス**: 効率的か
5. **テスタビリティ**: テスト可能な設計か

### セキュリティチェックリスト
- [ ] 入力値の検証
- [ ] SQLインジェクション対策
- [ ] XSS対策
- [ ] 認証・認可の適切な実装
- [ ] 機密情報の適切な処理

### 出力形式
- 重要度順の指摘事項（ファイル参照付き）
- テスト不足・カバレッジギャップの指摘
- 改善提案と修正例
